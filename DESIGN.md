# Design: Browser Breadboard Builder

Audience: you, building it yourself, to use for your studies

## Problem Statement

You want a tool to plan breadboard wiring for your electronics study: drop an
Arduino, some LEDs, resistors and buttons onto a virtual breadboard, draw the
jumper wires, and see the layout before you build it for real. You want to
build it yourself, and you want it to run in a browser with nothing to install.

You are not trying to replace all of Fritzing. You picked exactly one of its
four tools, the breadboard view, and exactly one edge, "runs in the browser."
Everything below is scoped to that and nothing more.

## What Makes This Cool

The payoff moment is small and specific: you grab a wire, drag it toward a hole,
the hole lights up, you let go, and the wire snaps in with a satisfying click of
feedback. Get that one interaction to feel good and the whole tool feels good.

Three things make this worth building over just using Tinkercad:

1. It is yours. You can shape it to exactly how your course works, add the exact
   parts your labs use, and you understand every line because you wrote it.
2. It is the most literal version of "nothing to install." A single page that
   runs from a URL or even a local file. No account, no load screen.
3. It teaches you the real web platform underneath: SVG, the DOM, pointer
   events, coordinate math, saving state. Skills that transfer to everything.

## Constraints

- Single-page web app, no backend required for v1.
- No build tooling required to run it (plain files, opened in a browser).
- Breadboard view only. No schematic, no PCB/Gerber export.
- No circuit simulation in v1. The tool draws the layout, it does not compute
  voltages or run code. (Simulation is where Tinkercad and Wokwi spend most of
  their effort. It is a separate, much larger project.)
- Built to be understood and extended by one person (you).

## Premises (agreed during the session)

1. Scope is the breadboard view only: place a small set of common parts, drag
   them onto a breadboard, draw jumper wires hole to hole, save the layout.
2. Platform is a static web app: plain HTML/CSS/JS rendering to SVG. Layouts
   save to the browser and to a JSON file. No server needed.
3. The hard, interesting 80% is the interaction model: a breadboard whose holes
   are electrically grouped like a real one, parts that snap to holes, and wires
   you draw between holes that feel good to place and delete.
4. Free browser breadboard tools already exist (Tinkercad Circuits, Wokwi).
   Building your own anyway is a deliberate choice: the goal is to create it and
   learn, not just to have access to one.

## Approaches Considered

### Approach A: Single-file vanilla SVG (CHOSEN)
One page, vanilla JavaScript, the breadboard is an SVG where each hole is a
clickable circle. Parts are SVG groups that snap to holes, wires are SVG paths
drawn hole to hole. Saves to localStorage and to a JSON file. Zero build step.

- Effort: Small. Risk: Low.
- Why it wins for you: your one stated edge is "runs in the browser, nothing to
  install," and a single SVG page is the most literal version of that. SVG holes
  are free to hit-test and highlight, so the core mechanic (snap a wire to a
  hole, light it up) is the easy part instead of the hard part. Best thing to
  learn from. You can graduate to a structured app later if it grows.

### Approach B: Structured app (Vite + Svelte + Konva canvas)
A real front-end project: Konva.js for the canvas scene, Svelte for the UI, a
clean data model, file import/export.

- Effort: Large. Risk: Medium.
- Why not now: build tooling and framework setup is friction before you reach
  the fun part. Overkill for v1. The right destination if the tool grows past a
  few dozen parts, not the right start.

### Approach C: Library-leveraged vanilla (Konva + reused open part art)
Vanilla JS but lean on Konva.js for drag/snap, and reuse Fritzing's open-source
part SVGs so you do not draw component art by hand.

- Effort: Medium. Risk: Medium.
- Why not now: fastest path to "looks real," but you learn less, you add a
  dependency, and you take on understanding Fritzing's SVG format and license.
  A good option if hand-drawing parts becomes the blocker, which it probably
  will not for the small v1 set.

## Recommended Approach: A, in detail

The rest of this doc is the actual design for Approach A.

### The breadboard electrical model (the heart of it)

A real breadboard is not a flat grid of independent holes. Holes are wired
together in groups underneath. Model that from day one, because it makes wiring,
saving, and any future "check my circuit" feature basically free.

A standard full-size breadboard:

- Main grid: 63 columns (an 830 tie-point board), rows labeled a, b, c, d, e in
  the top half and f, g, h, i, j in the bottom half. Within one column, the 5
  holes in the top half (a-e) are all connected to each other. The 5 holes in the
  bottom half (f-j) are all connected to each other. The top and bottom halves
  are separated by the center gap (the ravine). So each column holds two
  independent groups.
- Power rails: two rails at the top, two at the bottom. Each rail is a long run
  of holes that are all connected (a + bus and a - bus). Note: on many real
  full-size boards each rail bus is actually split into two halves at the middle,
  and wiring power to the wrong half is a classic beginner mistake. For v1, model
  each rail as one continuous group to keep it simple, but record this as a known
  divergence from real hardware: if you later build the wiring checker, switch to
  the split-rail model first, or the checker cannot catch that exact bug.

The clean way to represent this: every hole belongs to exactly one electrical
"node" (a group of connected holes). Define nodes once when you build the board:

- For each column c: node `col-top-c` = its a/b/c/d/e holes,
  node `col-bot-c` = its f/g/h/i/j holes.
- Rails: `rail-top-plus`, `rail-top-minus`, `rail-bot-plus`, `rail-bot-minus`.

Holes are then connected only by two things the user adds:
1. A component leg in a hole (the part bridges the nodes its legs sit in).
2. A wire between two holes (connects the two holes' nodes).

You do not need to act on that connectivity in v1. But because each hole already
knows its node, a future "is pin 13 actually connected to the LED, or did I make
a short across the power rails?" checker is just a union-find over nodes. Build
the model now, get the feature later for almost nothing.

### Geometry and coordinates

- Hole pitch (spacing) on a real breadboard is 0.1 inch. In SVG, pick a pitch in
  pixels, for example 20. All hole positions are computed from the pitch, never
  hand-placed.
- Main grid: column c sits at x = originX + c * pitch. Rows a-e at fixed y
  values, then a gap of about 2 pitches for the ravine, then f-j.
- Power rails sit above the top rows and below the bottom rows, each with its
  + line and - line.
- Each hole is an SVG `<circle>` (radius about 4) carrying data: a unique hole
  id (for example `T-12-c` or `RAIL-TOP-PLUS-7`), its node id, and its x/y.
- Keep a lookup table of all holes in JS: `{ id, x, y, node }`, plus a spatial
  helper to find the nearest hole to a point (for snapping).

### Data model (what gets saved)

Holes are deterministic, generated from the board definition, so they are NOT
saved. The save file only stores what the user added, referencing hole ids:

```json
{
  "version": 1,
  "parts": [
    { "id": "p1", "type": "led", "rotation": 0, "legs": ["T-12-c", "T-13-c"], "props": { "color": "red" } },
    { "id": "p2", "type": "resistor", "rotation": 0, "legs": ["T-10-c", "T-12-e"], "props": { "value": "220" } }
  ],
  "wires": [
    { "id": "w1", "from": "RAIL-TOP-MINUS-3", "to": "T-13-e", "color": "#1a1aff" }
  ]
}
```

This is small, human-readable, easy to diff, and trivial to export/import. It is
also the natural input to a future checker.

Validate on import: reject or warn if `version` is unknown, and drop (with a
visible message) any part or wire that references a hole id the current board
does not have, which happens if you load a full-size layout onto a half-size
board. Wrap localStorage reads and writes in try/catch: it can be disabled in
private-browsing mode or throw a quota error, and autosave should fail quietly
rather than break the page.

### Parts for v1

Keep the set small and study-relevant. There are two physical kinds of part:

Through-hole parts, whose legs plug into breadboard holes:
- LED (2 legs, has a + and - side, color property)
- Resistor (2 legs, value label like "220")
- Push button / tactile switch (4 legs). Note its internal wiring: the two pins
  on each side are permanently connected to each other, and pressing the button
  bridges the two sides. Model that internal connection in the part's data, not
  just four loose legs, so a future checker reads it correctly.
- (Optional) a basic IC / chip placeholder that straddles the center ravine

Boards, which have their own pins and sit beside the breadboard, with wires
running from their pins to breadboard holes:
- Arduino Uno, drawn as a rectangle with labeled header holes (D0-D13, A0-A5,
  5V, GND, etc). Wires attach pin to breadboard hole. Register the Arduino's pin
  holes in the same hole lookup table the wire tool uses, in the same board
  coordinate space, so a wire can connect a board pin to a breadboard hole with
  identical logic. The Arduino legs do not plug into the grid; only wires touch
  its pins.

Recommendation: ship the through-hole parts and wires first. Add the Arduino
board as a second milestone, because a board with its own pin map is a bit more
work, and once it lands "blink an LED" is fully buildable.

Define parts as data, not as one-off code. Each part type is an entry describing
its shape, its leg anchor points, its rotation behavior, and its SVG drawing.
Adding a new part then means adding a data entry, not writing new logic.

### Interaction design

Core actions:
- Add a part: click it in the palette, it appears attached to the cursor, drag
  it, its legs snap to the nearest valid holes on drop. Highlight target holes
  while dragging.
- Move a part: drag it, re-snap its legs.
- Rotate a part: R key or a rotate handle, in 90-degree steps.
- Delete a part: select and press Delete, or right-click.
- Draw a wire: pick the wire tool (or just click a hole), click the first hole,
  click the second hole, wire is drawn. A color picker sets wire color.
- Delete a wire: click it, press Delete.
- Save/Load: autosave to localStorage, plus export and import a JSON file.
- Export PNG of the layout for lab reports. This is v2, not v1. SVG-to-PNG via a
  canvas has real footguns (inline all styles first, watch for tainted-canvas
  security errors), so keep it out of the first build.

Snap logic (read this carefully, it is the one easy thing to get wrong). A part
with two or more legs has fixed spacing between its legs, so you cannot snap each
leg to its own nearest hole independently, or the part tears apart (leg 1 lands
in column 12, leg 2 jumps to column 15 because that happened to be nearest).
Instead: pick one leg as the anchor, snap the anchor to its nearest hole (within
about half a pitch), then derive the other legs' holes from the part's fixed leg
geometry and its current rotation. Light up all the target holes while dragging
so the user sees where the whole part will land, and reject (snap back) if any
derived hole is invalid.

A "valid" hole means it exists on the board and is not already occupied by
another component leg. The v1 policy: two legs may not share the exact same hole,
but legs and wire ends may share one, and reusing a hole in the same 5-hole node
is fine and expected, since sharing a node is the entire point of a breadboard.
To find the nearest hole, a plain linear scan over the hole table is fine. At a
few hundred holes it is fast, no spatial index needed.

Hit-testing is free here because every hole is a real DOM element. Pointer
events and `document.elementFromPoint` do the work that would be manual math on
a canvas.

### Rendering architecture

- One root `<svg>` with a `viewBox`. Pan and zoom by adjusting the viewBox, which
  is clean and keeps all the part/wire coordinates in board space.
- Layer the SVG into groups, back to front:
  1. board layer: the breadboard rectangle, the center ravine, the red/blue rail
     lines, column numbers and row letters, and all the holes (drawn once).
  2. wire layer.
  3. parts layer.
  4. overlay layer: hole highlights, selection outlines, the part being dragged.
- Holes are static, drawn once. Redraw only the wire and part layers from state
  when something changes. For a study layout (tens of parts, maybe a hundred
  wires) redrawing those layers on each change is plenty fast.

### State and undo

- Keep one `state` object: `{ parts, wires, selection, tool, view }`.
- A `render()` function rebuilds the wire and part layers from state.
- During a live drag, move only the dragged element for smoothness, then commit
  to `state` and call `render()` on drop. Do not re-render everything on every
  pointermove.
- Undo/redo: push a JSON snapshot of `{ parts, wires }` onto a stack on each
  committed change. Cheap to add, but it touches every action, so do not thread
  it through from the start. Add it in M5 once the core actions are stable.

### Build sequence (the order to actually build it)

M0. Render a correct breadboard. SVG with holes at the right positions, rails,
labels, the center gap. No interaction yet. Verify your electrical nodes are
assigned correctly by temporarily tinting each node group a different color, so
you can see the a-e and f-j columns and the rails group the way a real board
does. This is the foundation and the most satisfying first win.

M1. Wire tool. Click a hole, click another hole, draw a wire between them, store
`{ from, to }`, delete a wire, pick a color. This proves snapping and
hit-testing, which is the core of everything.

M2. First through-hole part: the LED. Palette, drag, snap two legs to two holes,
rotate, delete. Then generalize to resistor and button, since they follow the
same 2-leg / 4-leg, data-driven pattern.

M3. Save and load. Autosave to localStorage (wrapped in try/catch), export and
import JSON with validation on import. PNG export is explicitly v2, not here.

M4. Arduino board. A side board with labeled pin holes, wires attaching pin to
breadboard. Now "blink an LED" is fully buildable end to end.

M5. Polish. Pan/zoom, undo/redo, resistor value labels, nicer part art, keyboard
shortcuts, touch support via pointer events.

### Pitfalls and gotchas (the things that will actually bite you)

- Screen vs SVG coordinates. Pointer events give you screen pixels. You must
  convert them to SVG/board coordinates before snapping, or every position is
  subtly off. This is the single most common SVG bug. The robust recipe: make a
  point with `const pt = svg.createSVGPoint(); pt.x = e.clientX; pt.y = e.clientY;`
  then `const board = pt.matrixTransform(svg.getScreenCTM().inverse());`. Call
  `getScreenCTM()` on the same element consistently. (`createSVGPoint` works
  everywhere today but is officially superseded by `DOMPoint` plus `DOMMatrix` if
  you prefer the modern API.)
- elementFromPoint returns the topmost element. The part you are dragging and any
  highlight overlay sit on top of the holes, so `elementFromPoint` will hand you
  them instead of the hole underneath. Set `pointer-events: none` on the drag
  ghost and the overlay layer so the holes stay hittable. This is the classic
  gotcha that maps exactly onto this design's layer stack.
- Snap threshold. Store exact computed hole positions, use a threshold around
  half a pitch. Too small and snapping feels fussy, too large and legs jump to
  the wrong hole.
- Re-rendering during drag. Moving the whole scene on every pointermove will feel
  laggy. Move only the dragged element, commit on drop.
- Overlapping wires. Two straight wires on the same path are hard to tell apart.
  A slight curve or arc on each wire (the way Fritzing bows its wires) makes
  overlaps readable. Optional, but nice.
- The Arduino is a board, not a through-hole part. Its pins do not plug into the
  breadboard grid. It sits beside the board and wires run from its pins. Model it
  as its own thing with its own holes, do not force it into the grid.
- Pointer events, not mouse events, so it also works on a tablet or trackpad
  cleanly.

### File structure

"Single file is fine to start" is true, but the one split worth making early is
parts-and-board-as-data, which is what keeps the project from turning into
spaghetti. Suggested, with no build step (just `<script>` tags):

```
index.html      markup: the SVG container, the parts palette, the toolbar
style.css        styling
board.js         breadboard geometry + node generation (builds all the holes)
parts.js         part definitions as data (shape, legs, drawing)
app.js           interaction logic, state, render, save/load
```

Start in one file if you prefer, split out `parts.js` and `board.js` the moment
the single file starts to hurt. Do not over-structure on day one.

## Distribution Plan

- It is a static site, so hosting is free and trivial.
- Recommended: put it in a GitHub repo and turn on GitHub Pages. You get a URL
  you can open on any device and share with classmates, and the repo doubles as
  a portfolio piece.
- You can also just keep the `.html` file locally and open it directly, which is
  the absolute purest "nothing to install."
- No CI/CD needed for v1. GitHub Pages redeploys on push by itself.

## Success Criteria

v1 is done when, in a browser, with nothing installed, you can:
1. See a realistic breadboard with correctly grouped holes.
2. Drag an LED and a resistor onto it and have their legs snap into holes.
3. Draw colored jumper wires between holes and delete them.
4. Place an Arduino board and wire one of its pins to the breadboard.
5. Reload the page and your layout is still there, and export it to a JSON file.

If you can lay out a real "blink an LED" circuit in it and it looks right, it
works.

## Next Steps (the assignment)

Before writing any tool code, do this one concrete thing: open a real breadboard
(or a clear photo of one) and draw, on paper, the exact "blink an LED" circuit
you want to be able to build in the tool: the Arduino, the resistor, the LED,
and every wire, labeling which hole each leg and wire end goes into. That hand
drawing is your target. It pins down the precise board layout, the exact parts,
and the exact hole-grouping behavior the tool has to reproduce, and it becomes
your test case for M0 through M4. Build the tool to make that one drawing.

When that drawing exists, M0 (render a correct breadboard) is the first thing to
build, and it is the most satisfying win in the whole project.
